# SLO Check

Check SLO compliance and error budget status. $ARGUMENTS

## Process

1. **Read SLO definitions**:
   - Look for SLO definitions in:
     - `dev_docs/slos/` or equivalent project directory
     - Prometheus recording rules (generated by sloth or similar)
     - Infrastructure-as-code SLO definitions (Terraform, Pulumi)
     - Monitoring platform config (Datadog SLOs, Nobl9, Honeycomb)
   - For each SLO, extract:
     - Service name
     - SLI (Service Level Indicator): what's being measured
     - Target: the percentage (e.g., 99.9%)
     - Window: the time period (e.g., 30 days rolling)
     - Error budget: calculated from target and window

2. **Query current status**:
   - For each SLO, query the current SLI value:
     - Availability: `1 - (error_requests / total_requests)` over the window
     - Latency: `good_requests / total_requests` where good = under threshold
   - Calculate error budget consumption:
     - `budget_total = (1 - target) × window_duration`
     - `budget_consumed = (1 - current_SLI) × window_duration`
     - `budget_remaining = budget_total - budget_consumed`
     - `budget_remaining_percent = budget_remaining / budget_total × 100%`
   - **Signal**: SLI value, budget remaining %, burn rate

3. **Calculate burn rate**:
   - Current burn rate: how fast is the error budget being consumed RIGHT NOW?
     - `burn_rate = (error_rate_current / error_budget_rate) `
     - Where `error_budget_rate = (1 - target) / window_duration`
   - Burn rate interpretation:
     - 1x: consuming budget at exactly the sustainable rate
     - 2x: will exhaust budget in half the window
     - 10x: will exhaust budget in 1/10 of the window (emergency)
   - Project budget exhaustion date:
     - `days_remaining = budget_remaining / (burn_rate × daily_budget_rate)`

4. **Assess and alert**:
   - Apply alert thresholds:

   | Burn Rate | Budget Remaining | Alert Level | Action |
   |-----------|-----------------|-------------|--------|
   | < 1x | > 50% | HEALTHY | No action |
   | 1-2x | 25-50% | WATCH | Monitor closely |
   | 2-5x | 10-25% | WARN | Investigate contributing errors |
   | 5-10x | < 10% | ALERT | Active investigation required |
   | > 10x | any | PAGE | Immediate response needed |
   | any | 0% (exhausted) | BREACHED | SLO is violated, incident |

   - For elevated burn rates, identify contributing factors:
     - Which endpoints are generating errors?
     - Is it a specific error code (503 vs 500 vs 429)?
     - Is it affecting all users or a segment?
     - Did it correlate with a recent change?

5. **Recommend actions** based on budget status:
   - Budget > 50%: Business as usual. Safe to deploy risky changes.
   - Budget 25-50%: Proceed with caution. Smaller deploys, longer canary bakes.
   - Budget 10-25%: Slow down. Only critical fixes and well-tested changes.
   - Budget < 10%: Freeze non-critical deploys. Focus on reliability.
   - Budget exhausted: Incident mode. All hands on reliability until budget recovers.

6. **Produce report**:

```
## SLO Health Report
**Date**: YYYY-MM-DD
**Window**: [rolling period]

## SLO Status
| Service | SLI | Target | Current | Budget Total | Budget Remaining | Burn Rate | Status |
|---------|-----|--------|---------|-------------|-----------------|-----------|--------|

## At-Risk SLOs
| Service | Budget Remaining | Projected Exhaustion | Top Error Contributors |
|---------|-----------------|---------------------|----------------------|

## Recommendations
- [Service]: [specific recommendation based on budget status]

## Trend
- SLOs improving: N
- SLOs stable: N
- SLOs degrading: N
```

## Rules

- Error budget is the PRIMARY operational signal — not uptime percentage
- Burn rate determines urgency, budget remaining determines strategy
- A service with 0% error budget remaining is in INCIDENT mode by definition
- Burn rate alerts use multi-window detection: fast burn (5min + 1hr) for pages,
  slow burn (6hr + 3day) for tickets
- SLO targets must be achievable — a 99.999% SLO for a service that depends on
  a 99.9% external dependency is a lie
- Error budgets are not punishment — they're a tool for balancing reliability
  investment against feature development velocity
- When budget is low, the correct response is to slow down on features and invest
  in reliability — not to adjust the target
- Never change an SLO target without data justifying the change and stakeholder agreement
