# Data Quality Gates

## Principle
Every layer transition in an analytical pipeline is a **quality gate**.
Data that fails the gate is quarantined, not promoted. This is how you
maintain trust in the downstream data that powers dashboards and APIs.

```
Source → [GATE 0: Ingestion] → Raw layer
Raw    → [GATE 1: Clean]     → Cleaned layer
Cleaned → [GATE 2: Conform]   → Business/gold layer
Gold   → [GATE 3: Publish]   → Downstream (dashboards, APIs, external systems)
```

The specific layer names come from the active `platform-stack` skill. Common
conventions include **medallion** (bronze/silver/gold), **ELT** (staging/
transformed/marts), or **three-tier** (raw/clean/curated).

## Gate 0: Ingestion (Source → Raw)

**What**: Validate that raw data arrives in the expected shape.
**When**: On every ingestion run.

| Check | Type | Example |
|-------|------|---------|
| Schema match | Structure | All expected columns present |
| Primary key non-null | Completeness | `id` is never null |
| Row count bounds | Volume | Between 80%-120% of expected daily volume |
| Freshness | Timeliness | Latest record < 1 hour old |
| Source fingerprint | Lineage | Source system ID matches expected |

### Quarantine Pattern
Records that fail validation go to a quarantine table with failure reason,
rather than being silently dropped. Investigate in regular cadence.

## Gate 1: Cleaning (Raw → Cleaned)

**What**: Validate that cleaning transforms produce clean data.
**When**: On every cleaning materialization.

| Check | Type | Example |
|-------|------|---------|
| No duplicates | Uniqueness | `COUNT(DISTINCT id) = COUNT(*)` |
| Types correct | Type safety | `latitude BETWEEN -90 AND 90` |
| Tenant populated | Multi-tenancy | `tenant_id IS NOT NULL` |
| Referential integrity | Consistency | FK references exist in parent table |
| No data loss | Completeness | Cleaned count ≥ 95% of raw count |
| Timestamp monotonic | Ordering | `MAX(cleaned.ts) >= MAX(prev_cleaned.ts)` |

## Gate 2: Conforming (Cleaned → Business)

**What**: Validate that business aggregations and KPIs are correct.
**When**: On every business-layer materialization.

| Check | Type | Example |
|-------|------|---------|
| Aggregation correctness | Accuracy | `SUM(gold.count) = COUNT(cleaned.*)` per tenant |
| KPI in expected range | Sanity | `avg_fill_rate BETWEEN 0 AND 100` |
| All tenants represented | Completeness | Business layer has rows for every active tenant |
| Temporal consistency | Freshness | Date = today (or yesterday for daily) |
| No regressions | Stability | KPIs don't change > 50% day-over-day (anomaly) |

## Gate 3: Publication (Business → Downstream)

**What**: Validate that published data is consistent with the source.
**When**: After publish jobs + as API integration tests.

| Check | Type | Example |
|-------|------|---------|
| External format valid | Schema | Entity matches target system's expected schema |
| API response matches source | Consistency | API data = direct query result |
| Dashboard renders | Visual | Dashboard loads without error |

## Tooling

The specific asset-check/quality framework comes from the active `platform-stack`
skill. Common options:

- **Dagster asset checks** — native asset-graph integration
- **dlt schema validation** — catches drift at ingestion time
- **Great Expectations** — rich expectations, standalone
- **dbt tests** — SQL-based, integrated with transforms
- **Custom SQL assertions** — stored queries with pass/fail thresholds

## Test Data Strategy

### Fixtures for CI
- Small, deterministic datasets per vertical (10-50 rows per table)
- Stored in `tests/fixtures/` as Parquet or JSON
- Loaded into disposable test catalog

### Staging Environment
- Copy of production schema with anonymized/synthetic data
- Generated by a recurring job (e.g., weekly)
- PII masked before export from production

### Production Monitoring (not testing, but related)
- Asset checks run on EVERY production materialization
- Failed checks trigger alerts (PagerDuty, Slack, etc.)
- This is continuous quality monitoring — tests that never stop running
