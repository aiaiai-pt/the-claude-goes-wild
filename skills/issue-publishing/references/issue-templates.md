# Issue Templates by Tracker

## Markdown Format (Universal — always generated)

### Epic Issue

```markdown
# [Epic] M-{nn}: {Module Name}

> **Appetite**: {level} — {label} | **Type**: {solution_type} | **Team**: {size}

## Context
{2-3 sentences: what this module does, why it exists, who benefits.
Link back to the pitch problem statement.}

## Architecture
- **C4 Containers**: {list of containers this implements}
- **Key ADRs**: ADR-{nnnn} ({title}), ADR-{nnnn} ({title})
- **Full Spec**: [`docs/architect-process/architecture/modules/{nn}-{slug}/SPEC.md`]

## Versioning
- **Milestone**: {product} v{X.Y}
- **Component**: `{component-slug}`
- **Current Version**: {current or "new"}
- **Target Version**: {target}

### Conventional Commit Scope
All commits for this module use scope `({component-slug})`:
```
feat({component-slug}): add new capability
fix({component-slug}): handle edge case
```

## Scope

### Must-Haves
- [ ] F-{nn}.1: {title} — {1-line description}
- [ ] F-{nn}.2: {title} — {1-line description}

### Nice-to-Haves (~)
- [ ] ~F-{nn}.3: {title} — {1-line description}

### No-Gos (✗)
- ✗ {explicitly excluded thing}

## Dependencies
| Direction | Module/Feature | Status |
|-----------|---------------|--------|
| Blocked by | M-{nn} | ⏳ |
| Blocks | M-{nn} | — |

## Definition of Done
- [ ] All must-have features pass acceptance criteria
- [ ] API contracts validated (OpenAPI/AsyncAPI)
- [ ] Multi-tenancy conforms to active platform profile
- [ ] Observability: metrics + logs + traces present
- [ ] CI/CD: deployment manifests committed
- [ ] Security: scans passing in pipeline
```

### Feature Issue

```markdown
# F-{nn}.{m}: {Feature Title}

> **Module**: M-{nn} ({name}) | **Priority**: {must-have|~nice-to-have} | **Size**: {XS|S|M|L}
> **Component**: `{component-slug}` → v{target}
> **Version Bump**: `version:{component}:{patch|minor|major}`

### Commit Convention
```
{feat|fix}({component-slug}): {short description}

{longer description if needed}

Refs: F-{nn}.{m}
```

## Context
{Why this feature exists. Reference the user problem, not just technical need.}

## User Story
**As a** {role},
**I want to** {action},
**so that** {outcome}.

## Acceptance Criteria
- [ ] **Given** {precondition}, **when** {action}, **then** {result}
- [ ] **Given** {precondition}, **when** {action}, **then** {result}
- [ ] **Given** {error condition}, **when** {action}, **then** {error is handled gracefully}

## Technical Approach

| Aspect | Detail |
|--------|--------|
| **Service** | {container/service name} ({language/framework}) |
| **Database** | {tables/entities affected} |
| **Auth** | {identity + authorization references} |
| **Events** | {topic produced/consumed, or "none"} |
| **Frontend** | {component, or "none"} |

{2-4 sentences of opinionated implementation guidance, referencing patterns from
the active platform-stack profile.}

## API Contract
```http
{METHOD} /api/v1/{resource}
Authorization: Bearer {token}
```
{Request/response shapes or link to OpenAPI spec}

## Dependencies
- **Blocked by**: {F-nn.m, M-nn — or "none"}
- **Blocks**: {F-nn.m — or "none"}

## Out of Scope
- {What this issue explicitly does NOT cover}

## How to Verify
1. {Step}
2. {Step}
3. {Expected result}
```

## Linear-Specific Formatting

Linear uses Markdown. Use HTML comments for Linear project properties:

```markdown
<!-- Priority: Urgent | High | Medium | Low | No priority -->
<!-- Estimate: 0.5 | 1 | 2 | 3 | 5 | 8 -->
<!-- Labels: comma-separated -->
<!-- Cycle: {cycle name} -->

```

## GitLab-Specific Formatting

GitLab supports quick actions in issue descriptions:

```markdown
/label ~"appetite:big-batch" ~"type:mvp" ~"module:{slug}" ~"must-have"
/milestone %"{project_name}"
/weight {1-5 based on feature size}
/relate #{parent_epic_iid}

{Standard markdown body from template above}

---
_Auto-generated by architect process_
```

## GitHub-Specific Formatting

GitHub uses Milestones instead of epics and supports task lists:

```markdown
<!-- Milestone: {project_name} -->
<!-- Labels: appetite:big-batch, type:mvp, module:{slug}, must-have -->

{Standard markdown body from template above}

### Task Checklist
- [ ] Implementation
- [ ] Tests
- [ ] API docs updated
- [ ] Deployment manifest
- [ ] Code review

---
_Auto-generated by architect process_
_Parent: #{epic_issue_number}_
```
