# Issue Publishing Skill

## When to Use
- After module specifications are complete
- When the user wants to publish/sync issues to a tracker
- When re-publishing after spec updates

## Process

1. Detect available tracker CLIs:
   - `linear` CLI or `LINEAR_API_KEY` env var → Linear
   - `glab` CLI with auth → GitLab
   - `gh` CLI with auth → GitHub
   - Fallback: Markdown files (always available)

2. Ask the user to choose if multiple are available

3. Load `references/versioning.md` for SemVer, milestone, and conventional commit conventions

4. Create product milestone (representing the Shape Up cycle target)

5. Determine component versions for each module (new vs existing, bump type)

6. Read all module specs from `docs/architect-process/architecture/modules/*/SPEC.md`

7. For each module:
   a. Create an epic/parent issue with module context
   b. For each feature in the module, create a child issue with version labels
   c. Apply labels (appetite, type, module, component, version bump)
   d. Link dependencies between issues
   e. Assign all issues to the product milestone

8. Create RELEASE epic with release engineering setup tasks

9. Generate dependency graph as Mermaid

10. Save issue index + component-versions.md to `docs/architect-process/issues/`

## Label Taxonomy

### Required Labels
| Category | Labels |
|----------|--------|
| Appetite | `appetite:spike`, `appetite:small-batch`, `appetite:big-batch`, `appetite:multi-cycle` |
| Type | `type:dev`, `type:prototype`, `type:mvp`, `type:prod-mvp`, `type:production` |
| Priority | `must-have`, `nice-to-have` |
| Module | `module:{slug}` |
| Component | `component:{slug}` |
| Version | `version:{component}:{patch\|minor\|major}` |

### Optional Labels (profile-dependent)
When a platform profile is active, it may add profile-specific labels (e.g., a
domain taxonomy, a gap-tracking label). Load the active `platform-stack` skill
to see profile-specific label recommendations.

## Scripts

Execute the appropriate script based on tracker choice:
- `scripts/linear-publish.sh` — Linear API via curl
- `scripts/gitlab-publish.sh` — GitLab API via glab or curl
- `scripts/github-publish.sh` — GitHub API via gh CLI

Each script reads a JSON manifest of issues to create, generated by the issue-writer agent.

### Issue Manifest Format (intermediate JSON)

```json
{
  "project": "{project_name}",
  "tracker": "linear|gitlab|github|markdown",
  "milestone": {
    "name": "{product}-{version}",
    "due_date": "YYYY-MM-DD"
  },
  "components": [
    { "slug": "{component}", "current_version": "0.0.0", "target_version": "0.1.0" }
  ],
  "epics": [
    {
      "id": "M-01",
      "title": "Module: {name}",
      "description": "...",
      "labels": ["appetite:big-batch", "type:mvp", "module:auth"],
      "features": [
        {
          "id": "F-01.1",
          "title": "{feature title}",
          "description": "...",
          "labels": ["must-have", "module:auth", "version:auth:minor"],
          "blocked_by": [],
          "blocks": ["F-02.1"]
        }
      ]
    }
  ]
}
```

Save manifest to `docs/architect-process/issues/manifest.json` before publishing.

## Gotchas
- Linear uses GraphQL API — batch mutations for efficiency
- GitLab uses `/relate` quick action for issue linking
- GitHub doesn't natively support epics — use Milestones or Projects
- Always publish to Markdown as a backup, even when using a tracker
- Re-publishing should be idempotent — check for existing issues first
